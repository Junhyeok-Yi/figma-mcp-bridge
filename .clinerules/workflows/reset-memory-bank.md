# Reset Memory Bank Workflow

다음 디자인 프로젝트로 넘어갈 때 Memory Bank를 깨끗하게 비우는 워크플로우.
**브리지 도구의 구조(projectbrief, techContext 일부)는 유지**하고, **현재 디자인 작업과 묶인 컨텍스트만 리셋**한다.

## 언제 쓰나

- 이전 디자인 프로젝트(예: 메모 앱)가 끝나고 새 프로젝트(예: 결제 화면)를 시작할 때
- `activeContext.md` / `productContext.md` / `progress.md` 가 직전 작업으로 가득 차 있을 때
- 사용자가 **"reset memory bank"** 또는 **"새 디자인 프로젝트 시작"** 을 요청할 때

## 사전 점검

1. `git status` 로 커밋되지 않은 변경이 없는지 확인
2. `memory-bank/` 6개 파일을 한 번 빠르게 훑기
3. **이전 프로젝트의 결정사항·산출물 위치를 사용자에게 1줄 요약 보고**
   (예: "메모 앱 Desktop 1440px / Memo Board / Desktop 296:673 / DESIGN.md 토큰 적용")

## 파일별 처리

| 파일 | 동작 | 이유 |
|------|------|------|
| `projectbrief.md` | **그대로 유지** | 브리지 도구의 정의는 변하지 않음 |
| `productContext.md` | **새 프로젝트 정보로 교체** (사용자에게 1~3줄 요청) | 디자인 목표는 프로젝트마다 다름 |
| `activeContext.md` | **현재 초점만 1~2줄, 나머지 비우기** | 매 세션 갱신되는 살아있는 컨텍스트 |
| `systemPatterns.md` | **새 프로젝트의 화면/컴포넌트 구조 한 단락 + 비움** | 직전 프로젝트의 카드 구조는 더 이상 유효하지 않음 |
| `techContext.md` | **상단의 "도구 사용법" 유지, "현재 Figma 파일" 섹션만 교체** | 도구 사용법은 공통, 파일 정보만 갱신 |
| `progress.md` | **"완료/진행 중/미완료" 빈 골격만 남기기** | 직전 프로젝트 체크리스트는 종료 처리 |

## Step 1: 사용자에게 새 프로젝트 정보 요청

다음 세 가지를 한 번에 물어본다(누락 항목이 있으면 그 항목만 빈 골격으로 둔다):

1. 새 디자인 프로젝트 이름과 한 줄 설명
2. 현재 진행 단계 (탐색 / 와이어 / UI / 핸드오프 등)
3. Figma 파일·핵심 프레임 이름이 정해졌다면 그것

## Step 2: 파일 교체

위 표대로 파일을 갱신한다. 각 파일은 **1페이지 이내**로 유지한다.

`activeContext.md` 골격 예시:

```markdown
# Active Context

## 현재 작업 초점

- (한 줄: 무엇을, 어디까지)

## 다음 단계

- (3개 이내 bullet)
```

`progress.md` 골격 예시:

```markdown
# Progress

## 완료

(없음)

## 진행 중

(없음)

## 미완료

- (체크리스트 3~5개)
```

## Step 3: 리셋 결과 보고

사용자에게 다음을 1줄씩 보고한다:

- 갱신된 파일 목록
- 새 프로젝트 brief 한 줄
- 다음 액션 추천 (예: "DESIGN.md 검토부터 시작할까요?")

## 안 하는 일

- `DESIGN.md` 는 건드리지 않는다(디자인 토큰의 SSoT이므로 디자인 시스템이 정말 바뀐 경우에만 수동 수정)
- `.clinerules`, `.cline/skills/` 는 건드리지 않는다(브리지 도구의 동작 규칙)
- 코드/스크립트/템플릿은 건드리지 않는다
