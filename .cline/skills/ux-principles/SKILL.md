# UX Principles Guide

UI/UX 리뷰를 위한 평가 프레임워크.

## Nielsen's 10 Usability Heuristics

### 1. Visibility of System Status
시스템이 현재 상태를 적절한 피드백으로 알려주는가?
- 로딩 상태 표시 (스피너, 프로그레스 바)
- 현재 위치 표시 (네비게이션 활성 상태, breadcrumb)
- 작업 완료/실패 피드백 (토스트, 알림)

**검사**: 사용자가 "지금 무슨 상태인지" 항상 알 수 있는가?

### 2. Match Between System and Real World
사용자의 언어와 관습을 따르는가?
- 전문 용어 대신 일상 언어 사용
- 논리적 정보 순서 (자연스러운 흐름)
- 친숙한 아이콘과 메타포

**검사**: 비전문가가 즉시 이해할 수 있는가?

### 3. User Control and Freedom
실수를 쉽게 취소/되돌릴 수 있는가?
- 뒤로가기/취소 버튼
- 모달 닫기 (X 버튼, 바깥 클릭)
- 삭제 전 확인, Undo 지원

**검사**: "비상구"가 항상 보이는가?

### 4. Consistency and Standards
일관된 디자인 패턴을 따르는가?
- 동일 기능에 동일 UI 패턴
- 플랫폼 컨벤션 준수
- 색상, 타이포, 간격의 일관성

**검사**: 같은 것은 같아 보이고, 다른 것은 다르게 보이는가?

### 5. Error Prevention
오류를 미리 방지하는가?
- 입력 제약 (날짜 picker, 드롭다운)
- 위험한 작업 전 확인 (삭제, 결제)
- 실시간 유효성 검사

**검사**: 사용자가 실수하기 어렵게 설계되었는가?

### 6. Recognition Rather Than Recall
기억보다 인식에 의존하는가?
- 선택지를 보여주기 (드롭다운 > 직접 입력)
- 최근 항목, 제안 표시
- 라벨과 힌트 텍스트 제공

**검사**: 사용자가 아무것도 외울 필요가 없는가?

### 7. Flexibility and Efficiency of Use
초보자와 전문가 모두에게 효율적인가?
- 단축키, 고급 필터
- 개인화 옵션
- 빈번한 작업에 대한 빠른 경로

**검사**: 숙련된 사용자가 빠르게 작업할 수 있는가?

### 8. Aesthetic and Minimalist Design
불필요한 정보 없이 깔끔한가?
- 핵심 정보가 시각적으로 우선
- 불필요한 장식 최소화
- 적절한 여백과 그룹핑

**검사**: 화면에서 뺄 수 있는 요소가 있는가?

### 9. Help Users Recognize, Diagnose, and Recover from Errors
에러 메시지가 도움이 되는가?
- 구체적 오류 설명 (무엇이 잘못되었는지)
- 건설적 해결 방안 제시
- 기술 코드 대신 사람의 언어

**검사**: 에러 메시지를 읽고 사용자가 스스로 해결할 수 있는가?

### 10. Help and Documentation
도움말이 적절히 제공되는가?
- 복잡한 기능에 툴팁/가이드
- 검색 가능한 도움말
- 단계별 온보딩

**검사**: 막혔을 때 도움을 쉽게 찾을 수 있는가?

## 피드백 구조화 템플릿

```
[심각도] 카테고리 — 문제 설명
- 위치: 화면/섹션/구성요소 이름
- 현재: 현재 상태 구체적 설명
- 문제: 왜 이것이 문제인지 (사용자 관점)
- 제안: 구체적 개선 방안
- 근거: Nielsen Heuristic # 또는 UX 원칙
```

## 심각도 분류

| 심각도 | 정의 | 예시 |
|--------|------|------|
| **Critical** | 사용자가 핵심 작업을 완료할 수 없음 | 제출 버튼 누락, 필수 입력 필드 접근 불가 |
| **Major** | 사용성을 크게 저하시킴 (우회 방법은 있음) | 에러 메시지 부재, 혼란스러운 네비게이션 |
| **Minor** | 사소한 불편이나 비일관성 | 미세한 정렬 오류, 선택적 개선 |

## 우선순위 결정

1. **Critical** → 즉시 수정
2. **Major** → 다음 이터레이션에 반영
3. **Minor** → 백로그에 등록

동일 심각도 내에서는:
- 영향받는 사용자 수가 많은 것 우선
- 핵심 흐름(core flow)에 있는 것 우선
- 수정 비용이 낮은 것 우선 (quick win)
